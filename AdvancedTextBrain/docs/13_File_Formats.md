# 13. File Formats

## Purpose

The workflow is persistent-Markdown only. This chapter explains which Markdown files exist, what each one stores, and why internal tool transport must not be confused with a persistent project format.

## Persistent format policy

The Orchestrator may write only:

- `.spec-workflow/research.md`;
- `.spec-workflow/source-snapshot.md`;
- `.spec-workflow/draft.md`;
- `.spec-workflow/prior-spec.md`, only as a verified temporary rollback checkpoint for an existing publication;
- `spec.md`, after final acceptance.

The custom-agent definitions under `.github/agents/` and the linked execution contract under `.github/agent-contracts/` are also Markdown, but they are maintained by the project author rather than generated by a normal specification run.

No normal run persists executable source code, shell scripts, JSON, JSONL, databases, or downstream implementation files.

NEW cleanup is the only non-Markdown mutation: it may delete files of any extension under reserved `.spec-workflow/**` and `outgoing/**`. It never deletes human input or unknown paths.

## Custom-agent Markdown

Each `.agent.md` file has two parts:

1. **YAML frontmatter** configures the VS Code custom-agent name, description, target, tools, visibility, and allowed subagents.
2. **Markdown instructions** define behavior and contracts.

YAML frontmatter is VS Code custom-agent configuration. It is not runtime domain data and does not make YAML a workflow-state format.

The current Orchestrator is user-invocable and has read, search, edit, and subagent tools. Worker agents are not shown as normal user entry points and have read/search tools only.

The Orchestrator body links the separate execution contract using an ordinary Markdown link, which VS Code supports for reusing instruction files. The linked contract is part of framework behavior but is neither a fifth agent nor eligible task evidence.

## Source Markdown

Sources can use headings, prose, lists, tables, and ordinary Markdown links. The workflow reads their semantic requirements rather than requiring fixed filenames.

Source text can define legitimate output fields and structure. It cannot override the agent's roles, tools, allowed writes, evidence boundaries, or phase gates.

The current mixed-source test uses:

- [jira.md](../jira.md);
- [description1.txt](../input-spec/description1.txt);
- [description2.txt](../input-spec/description2.txt).

These are candidate files, not automatically one coherent requirement set. The goal/scope table can admit relevant sections and exclude incompatible embedded directives.

## Source-snapshot Markdown

`.spec-workflow/source-snapshot.md` contains one RUN_ID matching research, a sorted non-reserved candidate-path list, one `### SOURCE` block per candidate, logical line counts, and exact lines encoded as `L000001|text`. It exists to compare RESUME with current disk state. Candidate count equals path count and source-block count. For every block, `LINE_COUNT`, the number of contiguous rows, the highest row ID, and the editor-reported live line count must agree, and every row must match through EOF. It is excluded from task evidence and fully replaced on NEW; invented counts, timestamps, summaries, and model-generated hashes are not identity.

NEW creates and rereads `.spec-workflow/research.md` before this snapshot. Identity is valid only at the exact `.spec-workflow/source-snapshot.md` path; a root-level, basename-only, or duplicate current-run copy fails `MANAGED_ARTIFACT_PATH_CHECK`. A snapshot without matching research, or one that is misplaced, truncated, or structurally inconsistent, is uncommitted partial state. The Orchestrator repairs or cleanly retries NEW before calling a worker and never asks the user to update research or rerun baseline.

## Research Markdown

`.spec-workflow/research.md` is the audit and control record. It contains:

- current-run identity;
- run mode/reset reason, active goal/anchor, generated cleanup, managed-artifact path validity, source scope/snapshot, evidence membership, and single-run checks;
- included and excluded source inventory;
- three-layer Artifact Contract;
- `OBL-*` obligation register;
- Markdown Field Instantiation Contracts;
- expected-member ledgers;
- `IRQ-*` internal resolutions;
- `KQ-*` and `UNIT-*` coverage plan;
- field/member completion matrices;
- cross-unit calculations;
- publication manifest;
- exactly one ordered Run Invariant Block with reconciled gate, count, write, and success fields, including `CURRENT_REVISION_DEFECT_SIGNATURE`, `SAME_DEFECT_REVISION_COUNT`, `UNMAPPED_MANDATORY_OBLIGATION_COUNT`, `PRIOR_SPEC_STATE`, `PRIOR_SPEC_CHECKPOINT_STATE`, and `PRIOR_SPEC_CHECKPOINT_EQUALITY_CHECK`;
- `ACTION_AUTHORIZATION` records containing the action, tested field values, `GUARD_RESULT`, and reason;
- Phase Gate Ledger.

This file is deliberately not published. Internal classifications, worker notes, and forbidden-inference guidance belong here unless a current source explicitly requires them in the deliverable. A narrative status summary or session-only record cannot substitute for the complete durable baseline, invariant block, or action authorizations.

`PAUSED: HOST_AUTOPILOT_LIMIT` is a chat-only notice used only when VS Code explicitly reports the boundary for a session using Autopilot with `chat.autopilot.advanced.enabled: true`. It is never written for a Default Approvals tool-confirmation dialog and is not persisted as a gate value. Resumability comes from the active run ID, pending gate, accepted records, and durable Critic revision signature/count already stored in `.spec-workflow/research.md`.

## Draft Markdown

`.spec-workflow/draft.md` is the exact candidate reviewed by the Critic. It is not an outline and cannot be a partial batch dump.

The publication manifest records every draft block and its line range in `.spec-workflow/research.md`. If the draft changes, the manifest must be updated before final review.

## Published `spec.md`

Every accepted `spec.md` contains a `Specification Knowledge Base` with at least one source-grounded `KQ-*` entry.

The minimum Markdown envelope is:

| Element | Required content |
|---|---|
| `KQ-*` heading | Stable ID and source-specific title |
| `Question:` | Source-specific knowledge question, or the exact respondent-facing question when the native unit is question-native and the relationship is `IS_FINAL_ARTIFACT` or `EMBEDS_EXACT_UNITS` |
| `Answer:` | Concrete implementation knowledge, expected evidence, canonical response when required, or another valid response contract |
| `Source basis:` | One or more obligation IDs, readable labels, and verified plain locations |
| `Acceptance check:` | Observable verification condition |

Source-required fields are added inside the entry. The universal `Question:` and `Answer:` labels remain literal even when a source uses supplemental names such as `question_text`.

## Plain source locations

Persistent traceability uses:

- `path:line` for one line;
- `path:start-end` for an inclusive range.

Locations must be workspace-relative and verified from the current file read. Do not use:

- absolute filesystem paths;
- guessed line numbers;
- stale locations;
- Markdown hyperlinks as a replacement for the required plain location;
- compressed obligation ranges that hide individual labels.

Ordinary relative Markdown links are appropriate in documentation such as this chapter. They are distinct from the plain traceability-location contract inside generated specifications.

## Identifier namespaces

| Namespace | Meaning | Persistent location |
|---|---|---|
| `OBL-*` | Exact source-derived obligation | Research and published traceability |
| `IRQ-*` | Internal ambiguity-resolution query | Research only |
| `UNIT-*` | Exact source-native downstream unit when applicable | Research mapping and associated published KQ |
| `KQ-*` | Published Question → Answer knowledge entry | Research plan, draft, and `spec.md` |

Never reuse an `IRQ-*` as a published question. The Investigator creates published KQ wording independently during `PRODUCE_BATCH`.

## Markdown Field Instantiation Contract

The contract is a Markdown table or structured list at the smallest depth justified by the active sources:

- `CONTENT_AREA` records a required topic, its explicit constraints, source basis, and completion test when the source does not define per-item fields.
- `REPEATED_UNIT` records only source-required fields, dependencies, order, and source-supplied or source-derived cardinality for each repeated item.
- `ENUMERATION_OR_SCALE` expands only a finite set, range, or scale actually supplied by the source.

When no count or named coverage set is source-supported, durable contracts use `CARDINALITY: SOURCE_UNSPECIFIED`; do not invent a minimum. Expected-member and completion ledgers remain Markdown and appear only for applicable source-required members. The baseline records source-explicit shape facts; the full production contract is assembled after baseline and resolution acceptance, before Q&A production.

### Current cardinality spelling compatibility

The current implementation is not yet textually uniform. The Orchestrator's durable form is `CARDINALITY: SOURCE_UNSPECIFIED`; the Investigator's `BASELINE` response contract currently returns `SOURCE_CARDINALITY: UNSPECIFIED`; and the firewall refers to the value as `SOURCE_UNSPECIFIED`. These spellings represent the same “no source-supported count or named set” state. Documentation uses the Orchestrator form for persisted records and the Investigator form only when describing that worker's response. This note reports the current contracts; it does not claim that a normalization step exists.

## Internal JSON tool transport

VS Code and agent tools may internally send structured arguments or results that look like JSON. Examples include a subagent invocation request, a read-tool argument, or an edit operation description.

That transport is:

- ephemeral;
- owned by VS Code or the tool protocol;
- not source evidence;
- not a persistent workspace format;
- not copied into `.spec-workflow/research.md`, `.spec-workflow/draft.md`, or `spec.md` merely because it appeared in a tool call.

Seeing JSON-shaped tool transport does not authorize creating `.json` files.

## Optional JSON Schema

JSON Schema is not the default validation mechanism for this workflow.

Why:

- shape validation does not prove semantic completeness;
- a nonempty string can still contain `...`;
- a generic string can still hide a required enumeration as `1..N`;
- generated schemas can introduce unsupported fields or thresholds;
- persistent `.json` would violate the normal Markdown-only write boundary.

Use the Markdown Field Instantiation Contract and Critic checks by default.

If an active source or the user explicitly requires JSON Schema, represent it inside an allowed Markdown artifact when that satisfies the requirement, and still apply semantic completeness review. The current Orchestrator contract cannot create a `.json` file. If a separate persistent schema file is mandatory, report the write-policy conflict as blocked; changing that boundary requires an explicit framework-contract edit outside the run, not an ad hoc permission inside it.

## Relationship-specific completeness

File shape depends on the Artifact Contract relationship:

- `IS_FINAL_ARTIFACT` and `EMBEDS_EXACT_UNITS` require exact instantiated content.
- `DEFINES_SCHEMA` requires complete field, rule, dependency, applicability, and validation definitions; example values are required only when sources demand them.
- `REFERENCES_ONLY` requires verified references plus complete selection, access, interpretation, and use knowledge.

A schema or reference fails only when it substitutes for exact content that the source relationship requires.

## Failure modes

| Anti-pattern | Why it fails |
|---|---|
| Persisting tool-call JSON as workflow state | It mixes protocol transport with human-reviewable project artifacts. |
| Saving obligations as JSONL | It violates the Markdown-only boundary and fragments traceability. |
| Publishing the manifest | It leaks internal workflow structure into the domain specification. |
| Using `...` in a required list | The field exists syntactically but is not instantiated. |
| Writing `1..5` instead of five members | A compressed range does not satisfy an expected-member ledger. |
| Creating a JSON Schema to “fix” weak content | Schema shape cannot replace source-specific semantic checks. |
| Copying a prior `spec.md` into research | Prior output is not current-run source evidence. |
| Writing the snapshot before research, at the wrong path, or as a duplicate | The identity has no authoritative ledger or violates managed-path ownership. |
| Truncating a source block or inventing its count | The path/block/count/row/EOF/text equations cannot prove source identity. |

## Format checklist

- [ ] Every persistent generated file is Markdown.
- [ ] YAML is limited to `.agent.md` frontmatter.
- [ ] Tool transport is not copied into artifacts.
- [ ] Research-only records remain out of `spec.md`.
- [ ] NEW creates and rereads `.spec-workflow/research.md` before `.spec-workflow/source-snapshot.md`.
- [ ] The exact managed snapshot path is unique, with matching RUN_ID and no root-level or duplicate copy.
- [ ] Candidate/path/block counts and every per-block live-count/row-count/highest-ID/EOF/text equation pass.
- [ ] Orphaned, misplaced, or incomplete initialization is repaired locally before any worker call.
- [ ] Traceability uses verified plain relative locations.
- [ ] Every KQ has literal `Question:` and `Answer:` fields.
- [ ] Field and enumeration ledgers are Markdown-native.
- [ ] JSON Schema appears only under an explicit requirement and remains inside an allowed Markdown artifact; an external `.json` requirement is blocked under the current contract.
